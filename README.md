# kubectl-crd-sample

A kubectl plugin that generates an example YAML manifest for a
CustomResourceDefinition.

The plugin reads the CRD's OpenAPIv3 schema (storage version) from the API
server and walks it to emit a fully-formed `<group>/<version>` manifest, with
inline comments that surface each field's description, required/optional
status, and enum values.

## Usage

```sh
kubectl crd-sample <crd-name>
```

The CRD name is the fully-qualified CRD object name (`<plural>.<group>`).

Standard kubectl flags such as `--kubeconfig`, `--context`, `--cluster`,
`--user` and `--namespace` are accepted and honoured.

### Examples

```sh
# Print to stdout
kubectl crd-sample certificates.cert-manager.io

# Save to a file
kubectl crd-sample widgets.example.com > widget.yaml

# Use a non-default kube context
kubectl crd-sample widgets.example.com --context staging
```

### Sample output

For a `Widget` CRD with a string-enum `size`, an integer `port` with
`minimum: 1`, and a `tags` array with `minItems: 2`:

```yaml
# Generated by kubectl-crd-sample
# CRD:   widgets.example.com
# Kind:  Widget
# Scope: Namespaced
apiVersion: example.com/v1
kind: Widget
metadata:
  name: example
  namespace: default
# optional
spec:
  # Port the widget listens on.
  # optional
  port: 1
  # Replicas is the desired number of widget pods.
  # optional
  replicas: 0
  # Size selects the widget size.
  # required
  # enum: ["small", "medium", "large"]
  size: small
  # Tags is a free-form list of labels.
  # optional
  tags:
    - example
    - example
```

Sample values respect the schema's `minLength`, `maxLength`, `minimum`,
`maximum`, `exclusiveMinimum`, `exclusiveMaximum`, `multipleOf`, `minItems`,
and `maxItems`. `pattern` is **not** enforced — if a field declares a regex
pattern, the placeholder `example` may need to be tweaked by hand.

## How it works

1. Loads kubeconfig via `k8s.io/cli-runtime/pkg/genericclioptions`, so all
   the usual kubectl context/auth flags work.
2. Fetches the CRD with the typed `apiextensions-apiserver` clientset.
3. Picks the version whose `storage: true` flag is set.
4. Walks `openAPIV3Schema.properties` recursively, building a `yaml.Node`
   tree where every key carries a multi-line `HeadComment`:
   - field description (from the schema)
   - `required` or `optional`
   - `enum: [...]` (if applicable)
5. Encodes the node tree with `gopkg.in/yaml.v3`, which preserves comments.

`status` is omitted because it is a server-managed subresource; emitting it
would only confuse `kubectl apply`. `apiVersion`, `kind`, and `metadata` are
emitted with concrete placeholder values rather than being walked from the
schema.

## Limitations

- CRD v1 schemas only (`apiextensions.k8s.io/v1`). v1beta1 CRDs are not
  supported by the apiextensions-apiserver typed client.
- `oneOf` / `anyOf` / `allOf` constraints are not evaluated; every property
  declared on the schema is emitted, even if the schema only permits one
  branch at a time. After generation, prune the variants you don't want.
- `pattern` (regex) is not evaluated. Placeholder values may need editing
  to satisfy a CRD's pattern constraint.
- `x-kubernetes-validations` CEL rules are not surfaced in comments.
- Schemas exceeding 64 levels of nesting are truncated with a marker comment.
